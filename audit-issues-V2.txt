#1  [FIXED] CSS Injection via Unsanitized Calendar Colors in Inline Styles
    Files: calendarUtils.js (L242, L336-340), calendarLegend.js (L14),
           teamCalendarBoard.js (L3230)
    ────
    Calendar color values from server data are interpolated directly
    into inline style attributes without validation:

      colorBarStyle: `background:${calendarColor};`          (calendarUtils L242)
      `background:${color}`                                  (calendarUtils L340)
      `background:${row.rawColor};`                          (teamCalendarBoard L3230)
      `background:${color};`                                 (calendarLegend L14)

    A malicious color value like "#000;} .slds-modal{display:none" would
    break out of the style context. readableTextColor() already has a
    hex validation regex (/^#([0-9a-f]{3}|[0-9a-f]{6})$/i) that should
    be used as a gate before any color is interpolated into CSS.

    Fix: Add a sanitizeColor() function that validates against the hex
    regex and falls back to a default (#1b96ff) for non-matching values.
    Apply it in buildEventStyle(), colorBarStyle, and all inline usages.

#2  [FIXED] 4 Unhandled QueryException — Direct SOQL-to-SObject Assignment
    Files: GoogleCalendarConnectionService.cls (L304-309),
           GoogleCalendarSyncService.cls (L68, L124, L218),
           TeamCalendarSecurityController.cls (L139-143)
    ────
    These methods assign a SOQL result directly to a single SObject
    variable. If the record was deleted between page load and action,
    the query returns 0 rows and throws System.QueryException (not
    AuraHandledException), crashing the LWC with a generic error.

    Sites:
    a) GoogleCalendarConnectionService.getCalendarRow():
       return [SELECT ... FROM Team_Calendar__c WHERE Id = :calendarId LIMIT 1];

    b) GoogleCalendarSyncService.pushEvent():
       Calendar_Event__c evt = [SELECT ... WHERE Id = :calendarEventId LIMIT 1];

    c) GoogleCalendarSyncService.importEventsFromGoogle():
       Team_Calendar__c calendarRow = [SELECT ... WHERE Id = :calendarId LIMIT 1];

    d) GoogleCalendarSyncService.pushEventsForCalendar():
       Team_Calendar__c calendarRow = [SELECT ... WHERE Id = :calendarId LIMIT 1];

    e) TeamCalendarSecurityController.getUserCalendarAccess():
       User targetUser = [SELECT ... WHERE Id = :userId LIMIT 1];

    Fix: Convert each to the safe list pattern already used in
    RecordMutationService:
      List<SObject> rows = [SELECT ... LIMIT 1];
      if (rows.isEmpty()) {
          throw new AuraHandledException('Record not found.');
      }
      SObject record = rows[0];

#3  [FIXED] loadAllowedAssignedUser Treats null (No Restrictions) as "Current User Only"
    File: TeamCalendarEventPlannerController.cls (L204-220)
    ────
    When getAllowedSelectedUserIdsForCurrentUser() returns null, it means
    "no restrictions — all users are allowed." But this method converts
    null into an empty set, adds only the current user, then blocks any
    other assignedUserId:

      Set<Id> allowedUserIds = ...getAllowedSelectedUserIdsForCurrentUser();
      if (allowedUserIds == null) {
          allowedUserIds = new Set<Id>();    // wrong: null means "allow all"
      }
      allowedUserIds.add(UserInfo.getUserId());
      if (!allowedUserIds.contains(assignedUserId)) {
          return null;  // blocks availability check
      }

    This means any user without explicitly configured allowed-user IDs
    can only check availability for themselves, even though the intent
    is "allow all."

    Fix:
      if (allowedUserIds != null
          && !allowedUserIds.contains(assignedUserId)
          && assignedUserId != UserInfo.getUserId()) {
          return null;
      }

#4  [FIXED] Event Listener Memory Leak — calendarGrid & calendarAgenda
    Files: calendarGrid.js (L20-23), calendarAgenda.js (L17-20)
    ────
    Both components add 4 event listeners each in connectedCallback()
    using .bind(this) but have NO disconnectedCallback() to remove them.
    Each time the view switches (e.g., month -> week -> agenda -> month),
    the component is destroyed and re-created, leaking 4 listeners per
    cycle. After many view switches, dozens of orphaned handlers fire on
    mouseover/mouseout/mousedown/contextmenu, causing performance
    degradation and potential ghost event dispatches.

    calendarGrid connectedCallback:
      this.template.addEventListener('mouseover', this.handleNativeHoverOver.bind(this));
      this.template.addEventListener('mouseout', this.handleNativeHoverOut.bind(this));
      this.template.addEventListener('mousedown', this.handleNativeQuickActionMouseDown.bind(this));
      this.template.addEventListener('contextmenu', this.handleNativeQuickActionContextMenu.bind(this));

    calendarAgenda connectedCallback: (same 4 listeners)

    Fix: Store bound references in constructor, add disconnectedCallback:
      constructor() {
          super();
          this._boundHoverOver = this.handleNativeHoverOver.bind(this);
          // ... etc
      }
      disconnectedCallback() {
          this.template.removeEventListener('mouseover', this._boundHoverOver);
          // ... etc
      }

#5  [FIXED] Hover Card key={line} Uses Raw String — Duplicate Text Corrupts DOM
    File: teamCalendarBoard.html (L260-261)
    ────
    The hover card iterator uses key={line} where line is a plain string
    from hoveredEventPreview.lines. If two lines have identical text
    (e.g., two "All Day" or two blank lines), LWC gets duplicate keys,
    causing incorrect DOM patching and rendering corruption.

    Current: <template for:each={hoveredEventPreview.lines} for:item="line">
               <div key={line} ...>

    Fix: Change the data model so each line is an object with a unique
    index: { id: index, text: line } and use key={line.id}, rendering
    {line.text}.

#6  [FIXED] Dynamic SOQL Uses String Concatenation Instead of Bind Variables
    File: TeamCalendarBoardController.cls (L155-161)
    ────
    soql += ' AND Calendar__c = \'' + String.escapeSingleQuotes(calendarId) + '\'';
    soql += ' AND Status__c = \'' + String.escapeSingleQuotes(statusFilter) + '\'';

    While escapeSingleQuotes prevents injection, Salesforce best practice
    is to use bind variables in dynamic SOQL, which eliminates any
    injection risk entirely and is cleaner.

    Fix: Declare local variables and use bind syntax:
      Id resolvedCalendarId = (Id) calendarId;
      soql += ' AND Calendar__c = :resolvedCalendarId';

#16 [FIXED] All 5 Dialog Modals Missing aria-labelledby
    Files: calendarCreateModal.html (L9), calendarEventDrawer.html (L9),
           teamCalendarBoard.html (L269, L300, L384)
    ────

    <section role="dialog"> pointing to the heading <h2> via an id.
    Screen readers cannot announce the modal purpose without it.

    Fix: Add id="modal-heading-XX" to each <h2> and
