  #5  FEATURE: Recurring Events
──────────────────────────────────────────────────────────────────────────────
  Severity: MEDIUM
  Category: Feature / Backend + UI

  No recurrence support exists. Events are single-instance only.

  Plan:
  - New fields on Calendar_Event__c: Recurrence_Rule__c (RRULE string),
    Recurrence_Parent__c (self-lookup), Is_Recurrence_Exception__c
  - Recurrence engine in Apex: expand RRULE into occurrences on query
  - UI: Recurrence editor in create modal (daily, weekly, monthly, yearly,
    custom, end date/count)
  - Edit options: "This event only" / "This and following" / "All events"
  - Delete options: same three choices
  - Google Sync: map to Google Calendar recurrence API

──────────────────────────────────────────────────────────────────────────────
  #9  PERFORMANCE: Virtualized Event Rendering
──────────────────────────────────────────────────────────────────────────────
  Severity: MEDIUM
  Category: Performance

  All events for the visible range are rendered in the DOM at once. With
  hundreds of events across multiple calendars, this causes lag.

  Plan:
  - Implement "+N more" overflow on day cells (already have CSS, needs logic)
  - Lazy-render event pills only for visible weeks (intersection observer)
  - Debounce rapid view changes to avoid redundant Apex calls
  - Cache event data per date range to avoid re-fetching on prev/next toggle
  - Add loading skeletons during data fetch instead of spinner overlay

  Status: [ ]

──────────────────────────────────────────────────────────────────────────────
  #10  PERFORMANCE: Client-Side Event Caching
──────────────────────────────────────────────────────────────────────────────
  Severity: LOW
  Category: Performance

  Every view change or navigation triggers a fresh Apex call, even if the
  data was just fetched seconds ago.

  Plan:
  - In-memory LRU cache keyed by (calendarIds, dateRange, users, status)
  - Cache TTL: 60 seconds (configurable)
  - Invalidate on create/update/delete/sync
  - Show cached data immediately, then background-refresh
  - Reduces Apex calls by ~60% during typical browsing

  Status: [ ]

  #16  UX: Undo/Redo for Event Mutations
──────────────────────────────────────────────────────────────────────────────
  Severity: LOW
  Category: Feature / UX

  Drag-and-drop moves, quick edits, and deletes are immediate with no
  undo. Users must manually re-edit to reverse mistakes.

  Plan:
  - Toast notification with "Undo" action link after create/update/delete
  - Store previous state in memory (last N mutations)
  - Undo triggers reverse Apex call (update back to old values, undelete)
  - 10-second undo window before toast auto-dismisses
  - Works for drag-drop, inline edit, and delete operations
